# Yifei Labs 两周业务整改清单

> 文档日期：2026-07-10  
> 目标：在不大爆重写的前提下，把产品从「演示工具」推进到「可稳定部署、结果可解释、会话可回溯」的 MVP 基线。  
> 原则：**先保部署与资产，再提匹配质量，最后收紧漏斗体验。**

---

## 总览

| 周 | 主题 | 业务结果 |
| --- | --- | --- |
| **Week 1** | 可部署 + 可回溯 | Session 不丢；历史是完整职业会话；岗位库不再绑死本机文件 |
| **Week 2** | 更准 + 更闭环 | 方向预筛接上；分数可解释；匹配→诊断→计划主路径变强 |

每日可按 **0.5–1 人日** 估；若单人兼职，按优先级砍 P1 尾项。

---

## Week 1 — 部署硬伤与数据资产

### Day 1–2｜Session 持久化（P0）

**业务目标**：匹配 → 诊断 → 学习计划 共用同一 `sessionId`，进程重启 / 多实例后仍可续跑。

| 项 | 内容 |
| --- | --- |
| **改什么** | `src/lib/session/analysisSession.ts` 抽 `SessionStore`；新增 `supabaseSessionStore`；无 Supabase 时退回 memory 并 warn |
| **表** | `analysis_sessions`（id, resume_profile, match_result, gap_results, learning_plans, source, job_bank_version, client_token?, created_at, updated_at） |
| **API 形态** | `create/get/update/attach*` 改为 **async**；`actions.ts` 内 `await attach*` |
| **前端** | hooks 仍只传 `sessionId`；可选 localStorage 持久化 `sessionId` + `clientToken`，刷新可续 |
| **关联** | `useJobMatchAnalysis` / `useMarketFitAnalysis` / `useLearningPlan` / `JobMatchWorkspace` |

**验收**

- [ ] kill 服务后，用旧 `sessionId` 仍能 `get` 到 match
- [ ] 同 session 上 gap / plan 可追加，不新建孤儿 session
- [ ] 无 Supabase：功能不挂，仅内存 + 明确 warning
- [ ] 有 Supabase：生产路径默认写库

---

### Day 3｜历史「会话化」（P0）

**业务目标**：历史页名实相符——能打开一次完整职业分析，而不只是 match 快照。

| 项 | 内容 |
| --- | --- |
| **读** | 历史列表改读 `analysis_sessions`（或 session + 旧 `job_match_projects` 双读过渡） |
| **写** | match / gap / plan 成功后都 upsert session；`saveJobMatchProject` 可暂时保留作兼容 |
| **页** | `history/page.tsx`、`history/[id]/page.tsx`：展示 Top 岗、准备度、计划摘要、source |
| **弱身份** | 无登录：`client_token`（浏览器生成）过滤「我的档案」；管理端 service role 全量仍可用 |

**验收**

- [ ] 完成 match+gap+plan 后，历史详情三者都在
- [ ] 未配 Supabase 时空态文案准确（不承诺「已归档」）
- [ ] 旧仅 match 记录仍可读（兼容或标注「旧版」）

---

### Day 4–5｜岗位库可部署化（P0）

**业务目标**：生产环境岗位库可读、可增、实例一致；合成样本身份强制可解释。

| 项 | 内容 |
| --- | --- |
| **存储** | `jobDatabase.ts`：抽象 `JobBankRepository`；实现 `file`（dev）+ `supabase`（prod，`stored_jobs` 表或 JSONB 整包） |
| **字段补齐** | 读写强制 `isSynthetic: true`、补 `skillTags`（`normalizeStoredJob` 已有逻辑，**回填现有 JSON**） |
| **脚本** | `scripts/seed` 或一次性 migration：给现有 ~54 条打 `skillTags` / `isSynthetic` |
| **文案** | 工作台 / 结果卡 / 导出：固定「合成市场样本 · 非实时在招」；真实 JD 路径单独徽章 |

**验收**

- [ ] 两实例（或模拟两进程）读到同一岗位库版本
- [ ] 列表/匹配结果强制可见合成标识
- [ ] 本地无 DB 仍可用 file 模式开发
- [ ] `getJobBankStats` 与匹配用库一致

**建议表（示意）**

```sql
-- analysis_sessions
id uuid PK, created_at, updated_at,
resume_profile jsonb, job_bank_version text,
match_result jsonb, gap_results jsonb default '[]',
learning_plans jsonb default '[]',
source text, resume_text_hash text, client_token text

-- stored_jobs（或 job_bank_documents 单行大 JSON，先求稳）
id, title, company, location, salary, ...
skill_tags text[], is_synthetic bool default true,
batch_id, source, created_at, payload jsonb
```

## Week 2 — 匹配质量与漏斗闭环

### Day 6–7｜主匹配接上「方向」（P0/P1）

**业务目标**：不是整库乱撞，而是「先定方向再匹配」。

| 项 | 内容 |
| --- | --- |
| **修** | `useJobMatchAnalysis` 去掉写死 `directions: []` |
| **来源** | 用户选方向 / 先跑 `recommendRoleDirections` / 从 corpus 侧选择 tags |
| **后端** | `runMatchAgainstJobBankAction` 已支持 `directionTitles`，打通 UI 即可 |
| **可选** | 空 directions 时：用画像 `domains` 当弱方向，避免全库噪声 |

**验收**

- [ ] 选定「RAG 工程师」后，TopK 标题/relatedDirections 明显相关
- [ ] 不选方向仍能跑（降级全库 TopK）
- [ ] pipeline meta 可展示：方向过滤是否生效、候选数

---

### Day 8｜分数可解释（P1）

**业务目标**：用户 3 秒内知道「为什么这个分」。

| 项 | 内容 |
| --- | --- |
| **已有** | `scoreDimensions`、`ScoreDimensionsBar`、enrich blend |
| **补** | 每条角色固定展示：命中技能 / 缺失技能 / 经验与地点说明（短文案模板，不靠 LLM） |
| **约束** | LLM `matchScore` 只微调 overall，**不得抹掉**维度分 |
| **标识** | `source: ai \| fallback` + 合成样本 badge 同屏 |

**验收**

- [ ] Top3 每条都有四维 + 至少 1 条证据句
- [ ] fallback 与 AI 路径维度字段结构一致
- [ ] 导出 Markdown 含维度与合成声明

---

### Day 9｜画像与词表补强（P1）

**业务目标**：非「完美 NLP」，但明显减少「抽空技能 → 全员 50 分」。

| 项 | 内容 |
| --- | --- |
| **规则底** | 扩展 `skillOntology`（后端/数据/产品常见词）；项目抽取容忍更多中文标题 |
| **可选 LLM 层** | 仅当规则 `skillNames.length < N` 时调用轻量结构化；结果 cache by `rawTextHash` |
| **缓存** | 同 hash 简历不重复抽 profile（session 或简单表） |

**验收**

- [ ] 样例简历技能数稳定 ≥ 阈值
- [ ] 转行/产品类简历不再 skills=0（至少 domains 有值）
- [ ] profile 写入 session，诊断可复用（减少重复解析）

---

### Day 10｜漏斗收紧：匹配 → 诊断 → 计划（P1）

**业务目标**：少 mode 心智，多「下一步」。

| 项 | 内容 |
| --- | --- |
| **已有** | 一键诊断 inline、`CareerRouteBar` |
| **加强** | 匹配成功后主 CTA：「诊断 Top1」；诊断成功后主 CTA：「生成 30 天计划」 |
| **session** | optimize 也 `attach` 到 session（类型扩展 `optimizations?`） |
| **计划可用性** | `LearningPlanInput` 增加 `hoursPerWeek?`（默认 10）；UI 一个数字输入即可 |
| **历史** | 计划摘要进 session 详情 |

**验收**

- [ ] 不手动改 URL mode，也能走完 match→gap→plan
- [ ] 刷新后凭 sessionId（或历史详情）能看到三步结果
- [ ] 计划文案体现每周投入（有输入时）

### Day 11–12｜质量底线与产品诚实（P1/P2）

| 项 | 内容 |
| --- | --- |
| **黄金集** | 固定 10–20 份脱敏简历 + 期望方向；脚本跑 match Top1 是否合理（人工表即可） |
| **真实 JD 权重** | market-fit + 扩展导入路径在空态/引导中提升优先级；与「岗位库演练」并列入口 |
| **进度条** | 假进度改为对齐真实阶段文案：画像 → 预筛 → 模型 → 校准（仍可估时，但文案与 pipeline 一致） |
| **openai 拆分（可选）** | 若 Week1–2 有余力：只抽 `client/parse/completion` + match/gap 两个 service；**不阻塞业务验收** |

**验收**

- [ ] 黄金集跑完有表格记录（通过/存疑）
- [ ] 新用户首屏能分清：演练匹配 vs 真实 JD 诊断
- [ ] 分析中阶段文案与后端步骤大致对应

## 两周结束时的「完成定义」（DoD）

必须全部满足，才算本轮业务整改完成：

1. **Session 默认可持久**；多步分析不断档
2. **历史 = 会话**，至少含 match + gap + plan
3. **岗位库生产可读一致**；合成样本强制标识
4. **主匹配可按方向预筛**（不再永远 `directions: []`）
5. **分数带维度与证据**；AI/fallback/合成来源可见
6. **样例路径** 一键可演示：简历 → 匹配 → 诊断 → 计划 → 历史可回看
7. **无 Supabase / 无 Key** 仍可本地开发，且有降级提示

明确 **本两周不做**（避免范围膨胀）：

- 账号体系 / OAuth
- 向量检索 / Embedding
- 真实招聘站爬取
- `openai.ts` 全量拆完
- 学习计划打卡产品化
- 深色主题与动效大改

## 使用方式

- **评审时**：只争论 P0 是否进两周、表结构是否接受
- **开发时**：严格按 Day 顺序；Week1 未完不要开 Week2 体验项
- **验收时**：只用文末 7 条 DoD，避免「又改了一堆 UI 但 session 仍丢」

确认清单后，建议从 **Day 1–2 Session 持久化** 开工。
