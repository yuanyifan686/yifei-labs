# Yifei Labs 业务优化具体可落地方案

> 文档日期：2026-07-10  
> 输入来源：当前项目代码体检、`两周业务整改清单.md`、现有 `README.md` / `项目架构.md` / `前端架构设计.md`  
> 目标：把 Yifei Labs 从“可演示的 AI 求职工具”推进到“可部署、可信任、可复用、可持续迭代”的职业分析 MVP。

---

## 1. 一句话定位

Yifei Labs 不应被定义成泛泛的“AI 简历评分工具”，而应收敛为：

**面向真实岗位 / JD 的证据型简历匹配、能力差距诊断与行动计划工具。**

核心价值不是“AI 给你一个分数”，而是：

1. 告诉用户适合投什么岗位。
2. 解释为什么匹配或不匹配。
3. 指出简历证据和岗位要求之间的差距。
4. 给出下一轮投递前可以执行的优化动作。
5. 保存一次职业分析会话，支持持续迭代。

## 2. 当前项目判断

### 2.1 已经成立的部分

| 模块 | 当前状态 | 业务价值 |
| --- | --- | --- |
| 首页 | 已有 3D 简历扫描舱 + 岗位仪表盘方向 | 能建立“AI 职业分析”第一印象 |
| 工作台 | 已支持岗位匹配、能力诊断、学习计划、简历优化 | MVP 主链路完整 |
| 简历输入 | 支持粘贴和 PDF / DOCX / TXT / MD / CSV 上传 | 降低用户进入门槛 |
| 岗位库 | 本地 JSON 样本库，已有 54 条岗位 | 可支撑演示和初步匹配 |
| AI fallback | 无 API Key 时仍可生成规则结果 | 演示稳定性较好 |
| 报告 | 支持 Markdown 与打印 PDF | 有交付物感 |
| Chrome 扩展 | 可从 Boss / 智联导入 JD | 具备真实场景入口 |
| 构建质量 | `npm run lint`、`npm run build` 当前通过 | 基础工程健康 |

### 2.2 当前最大的业务问题

| 优先级 | 问题 | 影响 |
| --- | --- | --- |
| P0 | 历史记录会展示完整原始简历，但缺少登录、权限、隔离策略 | 不能公开上线 |
| P0 | 多步分析 session 只存在内存中 | 刷新、重启、多实例后链路断裂 |
| P0 | 岗位库仍依赖本地 JSON 文件写入 | 生产环境不可控，多实例不一致 |
| P0 | 岗位样本是合成市场样本，但数据身份没有强制展示 | 用户可能误以为是真实在招岗位 |
| P1 | AI 输出主要靠手写 JSON parse / normalize | 结果稳定性和可测试性不足 |
| P1 | 主匹配还没有把“目标方向”作为强约束打通 | 匹配结果可能噪声较大 |
| P1 | 缺少系统化测试：unit / e2e / visual | 每次 UI 或模型改动都可能回归 |
| P1 | 首页 3D canvas 在当前浏览器未稳定检测到 | 首屏科技感可能退化为 fallback |
| P2 | README 中“支持 PDF 导出”和“暂时不做 PDF 导出”并存 | 对外表达不一致 |

## 3. 业务落地总路线

建议分三层推进：

| 阶段 | 时间 | 目标 | 结果 |
| --- | --- | --- | --- |
| 第一阶段 | 0-2 周 | 修复 MVP 上线硬伤 | 可部署、可回溯、可解释 |

## 4. 第一阶段：两周可落地整改

### 总目标

两周结束时，产品必须能稳定完成：

**简历输入 -> 岗位匹配 -> 能力诊断 -> 30 天计划 -> 历史会话回看 -> 报告导出**

并且每一步都能说明：

- 数据来自哪里。
- 是否使用 AI。
- 是否使用 fallback。
- 岗位样本是否为合成。
- 分数为什么这样给。
- 用户下一步该做什么。

## 5. Week 1：部署、隐私、会话、数据资产

### Day 1-2：Session 持久化

#### 业务目标

让一次分析成为一个完整职业会话，而不是一堆孤立结果。

当前内存 session 会导致：

- 服务重启后结果丢失。
- 多实例部署后无法续跑。
- match / gap / plan 之间无法可靠串联。
- 历史页只能保存一段快照，不能代表完整分析过程。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| 抽象存储 | `src/lib/session/analysisSession.ts` 抽出 `SessionStore` 接口 |
| 默认实现 | `memorySessionStore` 继续保留，供本地和无 DB fallback |
| 生产实现 | 新增 `supabaseSessionStore` |
| API 调整 | `create/get/update/attachMatch/attachGap/attachPlan` 全部改 async |
| 前端保持 | hooks 仍只关心 `sessionId` |
| 刷新续跑 | 前端 localStorage 或 cookie 保存 `sessionId + clientToken` |

#### 建议表结构

```sql
create table analysis_sessions (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  client_token text,
  source text,
  resume_text_hash text,
  resume_profile jsonb,
  job_bank_version text,
  match_result jsonb,
  gap_results jsonb default '[]'::jsonb,
  learning_plans jsonb default '[]'::jsonb,
  optimizations jsonb default '[]'::jsonb
);
```

#### 验收标准

- kill dev server 后，用旧 `sessionId` 仍能恢复 match 结果。
- 同一个 session 可以追加 gap / plan / optimization。
- 无 Supabase 时功能不挂，但 UI 明确提示“仅本地临时会话”。
- 有 Supabase 时默认写入 `analysis_sessions`。

### Day 3：历史记录会话化 + 隐私隔离

#### 业务目标

历史记录必须从“匹配快照”升级为“完整职业分析档案”。

同时，历史记录涉及原始简历，是最高隐私风险点。公开上线前至少要做到弱身份隔离。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| 历史列表 | 改读 `analysis_sessions` |
| 历史详情 | 展示 match / gap / plan / optimization 摘要 |
| 弱身份隔离 | 浏览器生成 `client_token`，服务端按 token 过滤“我的记录” |
| 原始简历展示 | 默认不直接展示完整原文，只展示摘要 / hash / 可展开警示 |
| 旧表兼容 | `job_match_projects` 保留只读兼容，标记“旧版记录” |
| 空态文案 | 未配置 Supabase 时不承诺“已归档” |

#### 隐私底线

两周内可以不做 OAuth，但必须做到：

1. 不能让所有用户看到全量历史。
2. 不能通过猜 ID 直接读取他人简历。
3. 不能默认在页面上展开完整原始简历。
4. 需要在上传或粘贴处提示“简历会发送给 AI 服务用于分析”。
5. 需要提供“删除本次记录”或至少“本地不保存”开关的产品位置。

#### 验收标准

- A 浏览器生成的历史记录，B 浏览器默认看不到。
- 历史详情可以看到完整路径：匹配、诊断、计划。
- 原始简历默认折叠，并有隐私提示。
- 未配置 DB 时，历史页准确说明“未开启持久化”。

### Day 4-5：岗位库生产化

#### 业务目标

岗位库不能继续依赖本机 JSON 写入。生产环境要做到：

- 多实例读到同一版本。
- 数据来源可解释。
- 合成样本身份强制可见。
- 后续能引入真实 JD 和真实岗位数据。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| 抽象仓库 | `JobBankRepository`：`list/getStats/upsert/search` |
| 本地实现 | `fileJobBankRepository`，仅 dev 使用 |
| 生产实现 | `supabaseJobBankRepository` |
| 字段补齐 | 回填现有 54 条数据的 `isSynthetic: true`、`skillTags`、`batchId` |
| UI 标识 | 岗位卡、匹配结果、报告中统一展示“合成市场样本” |
| 版本 | 增加 `job_bank_version`，写入 session 和报告 |

#### 建议表结构

```sql
create table stored_jobs (
  id text primary key,
  title text not null,
  company text,
  location text,
  salary text,
  description text,
  requirements text,
  keywords text[],
  skill_tags text[],
  source text,
  batch_id text,
  is_synthetic boolean default true,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  payload jsonb
);
```

#### 业务文案统一

所有涉及岗位库的地方统一使用：

> 合成市场样本，非实时在招岗位。适合用于方向判断、能力对照和投递准备。

真实 JD 入口统一使用：

> 真实 JD 诊断，优先依据你提供的岗位描述。

#### 验收标准

- 本地 file 模式仍可开发。
- 生产 Supabase 模式不写本地文件。
- 岗位库统计与匹配使用同一数据源。
- 匹配结果和导出报告都能看到样本来源。

### Day 5：部署与降级 Checklist

| 场景 | 预期表现 |
| --- | --- |
| 无 API Key、无 Supabase | 可用 fallback 分析，不保存历史 |
| 有 API Key、无 Supabase | AI 分析可用，历史提示未开启 |
| 无 API Key、有 Supabase | fallback 可用，会话可保存，标注 fallback |
| 全部配置 | AI 分析、会话、历史、岗位库均正常 |
| AI 调用失败 | 自动 fallback，并提示结果可信度较低 |
| DB 写入失败 | 分析结果仍展示，但提示“未成功归档” |

## 6. Week 2：匹配质量、解释力、漏斗闭环

### Day 6-7：方向预筛打通

#### 业务目标

匹配不应是“拿简历去撞整个岗位库”，而应先判断方向，再在相关方向里排序。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| UI 入口 | 用户可选择推荐方向，或使用系统自动推荐方向 |
| 后端参数 | `runMatchAgainstJobBankAction` 使用 `directionTitles` |
| 弱方向 | 未选择方向时，用 profile domains / skill tags 作为弱过滤 |
| 展示 | pipeline meta 显示候选数、方向过滤是否启用 |

#### 验收标准

- 选择“RAG 工程师”后，Top 结果明显相关。
- 不选方向仍可跑全库 TopK。
- 匹配报告里能看到“从多少岗位中预筛 Top N”。

---

### Day 8：分数解释标准化

#### 业务目标

用户看到 72% 或 88% 时，必须马上知道原因。

#### 每个岗位卡必须展示

| 信息 | 来源 | 展示方式 |
| --- | --- | --- |
| 总匹配分 | deterministic + AI 微调 | 大分数 |
| 技能匹配 | `scoreDimensions.skills` | 条形或标签 |
| 经验匹配 | `scoreDimensions.experience` | 简短说明 |
| 方向匹配 | `scoreDimensions.direction` | 简短说明 |
| 地点匹配 | `scoreDimensions.location` | 简短说明 |
| 命中技能 | profile vs job | 绿色标签 |
| 缺失技能 | profile vs job | 黄色/红色标签 |
| 简历证据 | profile evidence | 1-2 条引用摘要 |
| 数据来源 | ai / fallback / synthetic | badge |

#### 产品原则

AI 可以参与表达和综合判断，但不能覆盖确定性维度分。  
fallback 和 AI 路径的数据结构必须一致。

#### 验收标准

- Top3 每条岗位至少有 1 条简历证据。
- fallback 结果也有完整维度分。
- 导出报告包含维度、证据、来源声明。

---

### Day 9：简历画像和技能词表补强

#### 业务目标

减少“简历抽不到技能 -> 匹配全员低分”的情况。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| 词表扩展 | 扩展 `skillOntology`：AI 应用、后端、数据、产品、运营、设计 |
| 项目抽取 | 支持更多中文项目标题，如“项目经历 / 实习经历 / 作品集 / 负责模块” |
| 弱画像 | 技能少时至少抽取 domains、项目方向、教育背景 |
| 可选 LLM | 当规则抽取技能数低于阈值时，调用轻量结构化抽取 |
| 缓存 | 按 `resume_text_hash` 缓存 profile |

#### 验收标准

- 技术类样例简历技能数稳定 >= 6。
- 产品/转行类简历不会出现 skills=0。
- profile 写入 session，后续 gap / plan 复用。

---

### Day 10：主漏斗收紧

#### 业务目标

用户不应该在 mode、tab、按钮之间迷路。产品主路径要像一条职业分析流水线。

#### 推荐漏斗

```text
上传/粘贴简历
  -> 开始岗位匹配
  -> 查看 Top 3
  -> 诊断最适合岗位
  -> 生成 30 天计划
  -> 优化简历
  -> 保存/导出/历史回看
```

#### 具体改造

| 位置 | CTA |
| --- | --- |
| 未上传简历 | “上传简历开始分析” |
| 匹配完成 | “诊断 Top1 岗位” |
| 诊断完成 | “生成 30 天计划” |
| 计划完成 | “优化这份简历” |
| 优化完成 | “导出报告 / 查看历史” |

#### 验收标准

- 不手动改 URL，也能走完整条链路。
- 每一步只有一个主 CTA，其他操作降级为次按钮。
- 刷新后能从 session 恢复当前进度。

---

### Day 11：真实 JD 权重提升

#### 业务目标

合成岗位库适合演练和方向判断，但真实 JD 才是用户真正要投递的对象。

#### 具体改造

| 项 | 落地内容 |
| --- | --- |
| 首页 CTA | 增加“粘贴 JD 做真实岗位诊断”入口 |
| 工作台 | JD 输入区域视觉优先级提高 |
| Chrome 扩展 | 当前通过 URL 携带最多 3500 字 JD，后续改为 token / storage / backend import |
| 结果解释 | 明确区分“真实 JD 要求”和“岗位库市场样本” |

#### 验收标准

- 用户能一眼分清“岗位库演练”和“真实 JD 诊断”。
- 诊断报告里真实 JD 的权重高于合成样本。
- 导入 JD 不再长期依赖超长 URL。

### Day 12：测试和验收体系

#### 业务目标

每次 UI、模型、岗位库、prompt 改动后，都能知道主链路有没有坏。

#### 建议测试分层

| 类型 | 内容 | 工具建议 |
| --- | --- | --- |
| Unit | 技能抽取、岗位预筛、分数计算、fallback normalize | Vitest |
| Contract | 固定 AI 返回 JSON，验证 normalize 后结构 | Vitest fixtures |
| E2E | 上传/粘贴简历 -> 匹配 -> 诊断 -> 计划 -> 导出 | Playwright |
| Visual | 首页 3D、工作台、技能星图、移动端布局 | Playwright screenshot |
| Smoke | 样例简历核心路径 | `npm run smoke` |

#### 需要加入的 npm scripts

```json
{
  "test": "vitest run",
  "test:watch": "vitest",
  "test:e2e": "playwright test",
  "smoke": "node scripts/smoke-core.mjs"
}
```

#### 验收标准

- CI 或本地至少能跑：`lint`、`build`、`test`、`smoke`。
- 首页检测到 canvas 或明确 fallback。
- 主链路 E2E 不依赖真实 API Key，可用 mock/fallback。

## 7. UI / 前端专项整改

### 7.1 首页 3D 简历扫描舱

当前方向正确，但需要补稳定性检查。

#### 问题

源码中存在 `THREE.WebGLRenderer`，但浏览器核对时未检测到 `canvas`，页面可能退回 fallback。

#### 落地动作

| 项 | 内容 |
| --- | --- |
| 运行检测 | 增加 dev debug：WebGL 支持、reduced motion、canvas mounted |
| 视觉兜底 | fallback 也要像“扫描舱”，不能只是静态卡片 |
| 性能 | 移动端降低粒子数，首屏延迟加载非关键元素 |
| 验证 | Playwright 检查 canvas 存在、尺寸非 0、截图非空 |

### 7.2 工作台技能星图

用户之前反馈技能星图“难看且看不清岗位需要的技能”。建议把它从“宇宙节点图”改成“岗位技能证据图”。

#### 新表达方式

| 区域 | 内容 |
| --- | --- |
| 中心 | 目标岗位 / Top 岗位 |
| 第一圈 | 必备技能 |
| 第二圈 | 加分技能 |
| 第三圈 | 缺口技能 |
| 右侧面板 | 点击技能后展示：岗位要求、简历证据、补齐建议 |
| 底部 | 技能覆盖率、缺口优先级、推荐行动 |

#### 原则

3D 不应牺牲可读性。分析页的可视化要服务决策，而不是服务炫技。

### 7.3 Button / Link 语义修复

首页存在 `Link` 包 `Button` 的嵌套交互元素。

#### 落地动作

| 方案 | 内容 |
| --- | --- |
| 推荐 | `Button` 支持 `asChild`，由 Link 承载交互 |
| 简化 | 把 Link 直接写成按钮样式 |

#### 验收标准

- DOM 中不出现 `a > button`。
- 键盘 Tab 顺序清晰。
- Playwright 点击 CTA 稳定。

### 7.4 右上角浅色点后字体颜色

之前反馈浅色点后的字体颜色不匹配。建议统一为：

| 状态 | 点颜色 | 文本颜色 |
| --- | --- | --- |
| AI 可用 | cyan / emerald | `text-cyan-100` 或 `text-slate-100` |
| fallback | amber | `text-amber-100` |
| 未配置 | slate | `text-slate-300` |
| 错误 | red | `text-red-100` |

避免浅色点后接过暗灰字，造成视觉断裂。

## 8. AI 可靠性专项整改

### 8.1 拆分 `openai.ts`

当前 `openai.ts` 同时承担：

- provider 配置
- prompt 调用
- JSON parse
- normalize
- fallback
- job generation
- match / gap / plan / optimization

建议拆成：

```text
src/lib/ai/
  client.ts
  completion.ts
  schemas/
    jobMatchSchema.ts
    skillGapSchema.ts
    learningPlanSchema.ts
  services/
    matchService.ts
    gapService.ts
    learningPlanService.ts
    resumeOptimizationService.ts
  fallbacks/
    matchFallback.ts
    gapFallback.ts
```

两周内不要求全拆完，但至少先抽：

1. `client.ts`
2. `safeJsonParse / completion.ts`
3. `matchService.ts`
4. `gapService.ts`

### 8.2 Runtime schema

用 Zod 或同类 schema 做运行时校验。

目标：

- AI 返回结构不完整时能修复或降级。
- 测试可以固定验证。
- UI 不需要防御一堆 undefined。

### 8.3 日志隐私

当前 parse 失败会打印模型输出片段。由于模型输出可能包含简历内容，建议：

- 生产环境只打印 requestId、label、错误类型。
- dev 环境最多打印截断结果。
- 不打印完整 resumeContent。
- session 中保存 `resume_text_hash`，避免日志里出现原文。

## 9. 数据与隐私策略

### 9.1 简历数据处理原则

| 数据 | 保存策略 |
| --- | --- |
| 原始简历文本 | 默认不长期展示；可配置是否保存 |
| 简历 hash | 默认保存，用于缓存和去重 |
| ResumeProfile | 可以保存，作为结构化画像 |
| AI 结果 | 可以保存 |
| 导出报告 | 用户主动生成 |

### 9.2 最低可上线隐私方案

两周内最低方案：

1. `client_token` 隔离历史。
2. 原始简历默认折叠。
3. 上传前有数据使用提示。
4. DB 中保存 `resume_text_hash`。
5. 提供“本次不保存历史”选项。
6. 服务端禁止无 token 全量读取历史。

中期方案：

1. Supabase Auth 或 OAuth。
2. `user_id` + RLS。
3. 删除记录。
4. 数据保留期限。
5. 敏感字段加密或脱敏。

## 10. 第二阶段：3-6 周发展方向

### 10.1 从岗位库演练转向真实 JD 闭环

重点做：

- JD 粘贴 / 扩展导入成为主入口之一。
- 用户可以保存多个目标 JD。
- 每个 JD 有独立匹配、差距、优化、投递状态。
- 合成岗位库只作为市场参考。

### 10.2 简历优化从“建议”变成“可接受 diff”

建议产品形态：

| 区域 | 内容 |
| --- | --- |
| 原文 | 用户简历原段落 |
| 建议 | AI 修改后的段落 |
| 理由 | 为什么改，匹配哪个岗位要求 |
| 风险 | 是否可能夸大经历 |
| 操作 | 接受 / 拒绝 / 手动编辑 |

底线：

- 不编造经验。
- 不虚构数据。
- 不把弱经历包装成确定成果。

### 10.3 职业项目管理

让用户回来继续用，而不是只分析一次。

功能方向：

- 投递项目列表。
- 每个岗位的准备度变化。
- 简历版本对比。
- 学习计划进度。
- 面试准备清单。
- 再次分析后的分数变化。

### 10.4 黄金集和质量评估

建立 10-20 份脱敏样例：

| 类型 | 用途 |
| --- | --- |
| AI 应用开发 | 主样例 |
| RAG 工程师 | 技术深度 |
| 产品经理转 AI PM | 转行样例 |
| 数据分析转 AI 应用 | 跨域样例 |
| 应届生 | 低经验样例 |
| 非技术运营 | 泛化样例 |

每份样例记录：

- 期望 Top 方向。
- 不应推荐的方向。
- 必须命中的技能。
- 必须指出的缺口。
- 人工评分。

## 11. 第三阶段：6-12 周商业化方向

### 11.1 推荐商业切口

优先切口：

**JD-specific resume match + evidence-based optimization**

也就是：

> 给我一个真实岗位 JD 和我的简历，告诉我能不能投、怎么改、先补什么。

这个切口比“泛职业规划平台”更容易成交，因为用户场景强、痛点近、结果可交付。

### 11.2 可尝试付费点

| 功能 | 付费可能性 |
| --- | --- |
| 单次真实 JD 深度报告 | 高 |
| 简历版本优化 diff | 高 |
| 多岗位对比 | 中高 |
| 30 天学习计划 | 中 |
| 历史追踪和版本对比 | 中 |
| AI 聊天咨询 | 中低，容易同质化 |

### 11.3 产品包装

建议落地成三种入口：

1. **免费快速匹配**：粘贴简历，获得岗位方向和 Top3。
2. **真实 JD 诊断**：粘贴 JD，获得准备度和缺口。
3. **投递前优化报告**：输出可修改简历建议和行动计划。

## 14. 完成定义

### 两周 MVP DoD

必须全部满足：

- [ ] Session 默认可持久化。
- [ ] 历史页展示完整会话，而不是孤立 match 快照。
- [ ] 历史记录有弱身份隔离，不能全局公开读取。
- [ ] 岗位库生产环境不依赖本地文件写入。
- [ ] 合成岗位样本在 UI 和报告中强制标识。
- [ ] 主匹配支持方向预筛。
- [ ] Top3 岗位有维度分、命中技能、缺失技能、证据句。
- [ ] fallback 和 AI 结果结构一致。
- [ ] 主路径可一口气完成：简历 -> 匹配 -> 诊断 -> 计划 -> 历史。
- [ ] `lint`、`build`、核心 smoke 通过。

### 不纳入两周范围

为了避免范围膨胀，两周内明确不做：

- 完整账号体系 / OAuth。
- 支付。
- 向量检索 / Embedding。
- 真实招聘站自动爬虫。
- 学习计划打卡产品化。
- `openai.ts` 全量重构。
- 大规模 UI 主题重做。

## 15. 风险与处理

| 风险 | 处理方式 |
| --- | --- |
| Supabase schema 一次改太多 | 先新增 `analysis_sessions`，旧表只读兼容 |
| client_token 不是强安全 | 两周内作为弱身份隔离；公开商业化前上 Auth + RLS |
| AI 输出不稳定 | fallback 结构统一 + runtime schema + contract tests |
| 岗位库合成身份影响信任 | 主动说明，用真实 JD 入口承接高信任场景 |
| 首页 3D 性能影响首屏 | 延迟加载、移动降级、canvas 检测 |

## 16. 最推荐的下一步

不要从 UI 继续大改开始。

建议马上从这三件事开工：

1. **SessionStore + Supabase session**  
   这是所有历史、闭环、复用的地基。

2. **历史页隐私隔离**  
   原始简历属于敏感数据，这个不修不适合公开使用。

3. **岗位库合成标识 + 数据源抽象**  
   这能同时解决部署和产品诚实问题。

完成这三件事后，再做方向预筛、分数解释和主漏斗优化，产品会从“好看的分析工具”变成“能被真实用户信任的职业分析 MVP”。
